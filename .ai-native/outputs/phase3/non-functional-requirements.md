# Phase 3: 非機能要件定義

- **作成日:** 2026-07-15
- **作成ロール:** 壁打ちナビゲーター
- **前提:** 本フェーズ群の成果物は**モックアップ**。本番実装時に引き上げる項目は「将来」列に記載する。

## 1. パフォーマンス

| 項目 | モックアップ基準 | 将来（本番） |
|---|---|---|
| 初期表示 | ページ遷移後 200ms 以内に主要コンテンツ描画（SPA 内遷移） | 同等（review-standards LAYER_4 準拠） |
| 検索・フィルタ | 100ms 以内に反映（クライアント内計算） | サーバー集計 + 体感優先 |
| モックデータ量 | 一覧表示が体感できる件数（メンバー 12 名 / 顧客 8 社 / 3 ヶ月分の業務データ目安） | 実データ規模 |

## 2. ユーザビリティ（U-1〜U-5 準拠）

| ID | 要件 |
|---|---|
| U-1 | 入力は選択肢化・プリフィル・0.25h 刻み等で最小化。必須項目を明示 |
| U-2 | 集計値は単位・前年比等の文脈付き。状態はバッジ色で直感表示（正常=緑/警告=黄/重大=赤/情報=青） |
| U-3 | 主要業務（打刻→日報→申請）が 2 クリック以内で開始できる |
| U-4 | 破壊的操作は確認ダイアログ。エラー・警告は原因と次アクションを文言に含む |
| U-5 | フォーカスリング可視・ARIA ロール（タブ・ダイアログ・ライブリージョン）・コントラスト AA を共通コンポーネント側で担保 |

## 3. データ機密度分類

| 区分 | データ | 扱い |
|---|---|---|
| C3（最高機密） | 勤怠・有給・給与関連、日報原文、エスカレーション内容、監査ログ | 本番では管理者/本人のみ。原文を返さない集計 API 設計（ai-manager 方式）。モックでは擬似ユーザー切替で境界を体感 |
| C2（社外秘） | 顧客マスタ・顧客関係・ナレッジ・プロジェクト・売上 | ロール別アクセス制御。分析側は access_policy で配信制御 |
| C1（社内公開） | 稼働状況・掲示情報・外部リンク | 全メンバー閲覧可 |

## 4. セキュリティ（モック段階の方針）

- 認証は擬似ログイン（ユーザー切替）で代替。本番は SSO（Google Workspace）前提の Provider 差し替え設計
- シークレットのハードコード禁止（AS-4）。リポジトリにトークン・鍵をコミットしない
- XSS: `v-html` を使用しない。リッチ強調はセグメント型（undeux `RichSegment` 方式）で描画
- 外部リンクは `rel="noopener noreferrer"` を必須とする

## 5. 可用性・運用（将来要件の宣言）

- 冪等性: 打刻・提出・承認は二重操作で重複レコードを作らない（モックでも状態機械で担保）
- 記録系データ（打刻履歴・承認ログ・活動ログ）は再実行・再生成で巻き戻さない（開発原則 2）
- 補助処理（通知・ナレッジ還流）の失敗が主フローを止めない（開発原則 4）
- エラーコード体系: `AKO-{領域}-{連番}`（例: AKO-ATT-001）を本番実装で導入。モックではトースト文言に一貫性を持たせる

## 6. 互換性・移植性

- 対応ブラウザ: 最新版 Chrome / Edge / Safari。画面幅 360px〜1920px
- モックデータ層（composables の I/F）は将来の REST API 差し替えを前提に、呼び出し側と分離する（phase5/api-design.md）
- 他社展開: 文言・選択肢・カスタム項目・機能トグル・外部リンクを設定データ化し、コード変更なしで適応できる範囲を最大化する

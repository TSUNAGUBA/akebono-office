## 0. 背景と位置づけ

### 0.1 何を作るか（1 文）

**業界・業種に応じた標準構成で商品マスタ〜請求までの業務アプリ群を提供し、管理者の設定（使用アプリ・業態・項目カスタマイズ）だけで自社にも他社にも適応できる「Akebonoメニュー」を akebono-office 上に構築する。**

- F-03 AKEBONO（現在は「要件定義中」プレースホルダ + 要望ボックス）の**第一弾の実体**である
- 現行の売上管理（F-15 `/sales`）は本アプリ群へ**移管**する
- 真の課題（phase1/true-problem.md）の「業務データがスタースキーマに蓄積され分析・意思決定支援に利活用できる」「本アプリ自体が他社向け商材になる」の実現手段そのものであり、既存の勤怠・日報等（社内業務）に対して**商流業務（SCM/販売管理）**のデータをAI文脈・分析基盤へ合流させる

### 0.2 参考リポジトリとの関係（調査済み 2026-07-20）

| 観点 | 内容 |

## 2. ユースケース増分（UC-09〜UC-13）

既存 UC-01〜08 に追加する。いずれも「入力 → 処理 → 出力」で記述。

### UC-09 陶磁器委託販売の月次サイクル（自社・最重要）

- **入力:** 作家（仕入先）からの陶磁器仕入登録 → 店舗（得意先）への納品（出荷登録）→ 店舗からの月次売上報告（CSV 取込 or 手入力）
- **処理:** 仕入 → 在庫増、納品出荷 → 店舗預け在庫へ移動、売上報告 → 売上明細計上（事業セグメント = 陶磁器）→ 月締めで精算エンジンが**店舗向けマージン請求**と**作家向け支払明細**を委託条件マスタ（手数料率）から自動生成
- **出力:** 店舗別売上・請求書（ドラフト → 発行）・作家別支払通知・セグメント別売上サマリ
- **例外:** 売上報告の訂正 → 赤黒（マイナス明細）で再計上。発行済み請求は不変、訂正は赤伝

### UC-10 業態設定でメニューが決まり、カスタマイズできる

- **入力:** 管理者が自社の事業セグメント（業態）を登録（例: 陶磁器委託販売 = 小売系、SI 事業 = 情報サービス系）
- **処理:** 業種プリセット（§3.3）から使用アプリの既定が展開される → 管理者が使用/不使用をトグルで調整（例: 小売プリセットに出荷管理を追加）
- **出力:** 使用 = ON のアプリのみが Akebonoメニュー（ダッシュボード・`/akebono` ハブ）に表示される。全メンバーに即反映
- **例外:** 実績データが存在するアプリを不使用化 → メニュー非表示になるがデータは保全（再有効化で復元 = 開発原則 2・9.5）

### UC-11 SI/SaaS 売上の計上とセグメント別照会

- **入力:** SI 案件・SaaS の売上明細（買い切り/固定月額/従量課金の課金区分付き）
- **処理:** 商品マスタ（サービス品目）から派生した売上明細としてセグメント = SI/SaaS で計上。従量課金は数量 × 単価
- **出力:** 売上管理でセグメント横断の合算ビューと、セグメント絞り込みビューを切替・並列比較できる
- **例外:** 外注を使う SI 案件は発注管理（外注費）・仕入管理（外注納品受入）に読み替えた画面文言で運用する

### UC-12 既存システムからのデータ移管（取込マッピング）

- **入力:** 既存システムの商品・取引先・売上データ（CSV / 固定長テキスト / JSON ファイル、または API 接続）
- **処理:** 取込元定義 → 項目マッピング（AI 候補提示 + **人が確定**）→ 変換（コード変換・型変換・正規化）→ 検証（DQ チェック）→ dry-run プレビュー → 反映（自然キー upsert = 冪等）
- **出力:** 取込実行履歴（件数・エラー行明細）・マッピング定義（版管理・再利用可能）
- **例外:** エラー行は隔離して健全行を止めない（行単位グレースフルデグラデーション = 開発原則 4）

### UC-13 アパレル業での SKU 展開（他社展開の代表シナリオ）

- **入力:** 商品（親）にカラー × サイズの SKU 展開を登録。画像はサンプル画像（企画段階）→ 製品画像（本番）を複数登録
- **処理:** 発注・入荷・在庫・出荷・売上は **SKU 単位**で管理。SKU マトリクス UI（縦 = カラー、横 = サイズ）で一括入力
- **出力:** SKU 別在庫・SKU 別売上。商品単位へのロールアップ表示
- **例外:** SKU 展開のない業種（陶磁器の一点物・SI サービス等）は既定 SKU 1 件を自動生成し、画面上は SKU を意識させない

### 3.2 業態（事業セグメント）モデル

「どんな業態を扱っているかを設定できるメンテナンス機能」の実現:

- **事業セグメントマスタ（F-30）**: 自社が営む業態を複数登録できる（例: `陶磁器委託販売`・`SI`・`SaaS`）。各セグメントは**業種タイプ**（`retail` 小売 / `maker` メーカー / `logistics` 物流・倉庫 / `it_service` 情報サービス / `other`）を 1 つ持つ
- **全トランザクション（発注・仕入・入出荷・売上・請求）はセグメント ID を持つ**。売上管理・請求管理は「全セグメント合算」と「特定セグメント絞り込み」を切替でき、セグメント並列比較ビューを持つ（UC-11）
- 既存 `SalesMonthly.projectType`（biz_consulting/sys_consulting/development/operation/internal）は事業セグメントへ**写像して移行**する（設計書 §9）
- **設計判断（業界と業種の関係）:** 既存の業界マスタ（F-10-2 Industry。顧客・自社の属性）はそのまま**分析・AI 文脈の軸として継続**する。メニュー・項目の既定を決める軸は業種タイプ 5 値（industryType）に**集約**し、「業界 × 業種」の 2 段プリセットは持たない（業界差はカスタマイズ = F-20-3/F-31 で吸収する）
- セグメント数の想定: 自社 2〜5。他社展開でも 10 以下を想定（超える場合は運用を見直す）

### 3.4 業種による項目・文言の差（基本項目セット）

- 各アプリのフォーム入力項目・一覧表示項目には**業種タイプ別の基本項目セット**がある（例: アパレル小売 = カラー/サイズ列が既定 ON、情報サービス = 課金区分が既定 ON、在庫系項目は既定 OFF）
- 基本項目はカスタマイズ（F-31）で**差し引き（非表示化）**・追加・必須化・並び替えができる。基本項目の削除は不可・非表示化のみ（scm-platform ADR-039 と同旨 = 将来の再表示・データ保全）
- 情報サービス業の「発注 = 外注費/工数」「仕入 = 外注納品受入」のような**読み替えは、アプリ表示名・項目ラベルのオーバーライド**で実現する（コード分岐を作らない）

## 6. 非機能要件（増分）

既存 non-functional-requirements.md を継承し、以下を追加する。

### 6.1 データ量想定（決定 #14・2026-07-20 で確定）

| データ | 自社想定 | 他社展開時の設計上限 |
|---|---|---|
| 商品 / SKU | 〜3,000 / 〜10,000 | 50,000 / 500,000 |
| 売上明細 | 月 〜5,000 行 | 月 500,000 行 |
| 在庫トランザクション | 月 〜10,000 行 | 月 1,000,000 行 |
| 取込 1 回あたり | 〜10,000 行 | 100,000 行（超過は分割案内） |

### 6.2 パフォーマンス

- 一覧 API: フィルタ適用済み 1 ページ（既定 50 件）の応答 300ms 以内（p95）。全件ダンプ API は提供しない（XA-6）
- 在庫残高照会: 残高キャッシュ参照で 200ms 以内。キャッシュ再構築（手動回復パス）は非同期実行
- 取込: 10,000 行を 60 秒以内（ステージング + 検証）。反映はトランザクション分割（1,000 行単位）で部分成功を報告

### 6.3 機密度（既存 3 区分に追加）

| 区分 | データ |
|---|---|
| C2（社外秘） | 商品原価・仕入価格・マージン率・委託条件・請求/支払金額・取引先条件（既存 C2 と同水準。F-16 の機能 deny 既定: member は請求管理・仕入管理 deny を提案 = 運用デフォルト F-16-4 の増分） |
| C1 | 商品名・画像・カテゴリ等の商品基本情報（社内公開） |

### 6.4 可用性・整合性

- 在庫残高・月次売上・請求ドラフトは**導出データ**であり、SoT（トランザクション台帳・売上明細）から常に再構築可能にする（開発原則 6）
- 取込・締め・ETL はすべて冪等（再実行で二重計上しない）。冪等キーは設計書 §3 で宣言
- 請求発行済みデータは不変（確定系）。会計監査に耐える証跡（採番・発行日時・発行者・赤伝リンク）

## 7. ユーザー権限

- 既存 F-16（3 レイヤ権限ルール）に Akebonoメニューの機能キー（akebono-products / akebono-inventory / akebono-billing 等）と表示項目制御（原価・マージン率）を追加する

- アプリ使用/不使用（F-20-3）は「テナントとしての導入状態」、F-16 は「誰が使えるか」。**両方 OK のときのみメニュー表示・API 許可**（scm-platform の tenant_application × entitlement の 2 段機構と同型）
